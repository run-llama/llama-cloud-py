# Research: SDK Documentation Surfaces

**Date**: 2026-03-11
**Commit**: 4795f3e
**Branch**: eli/extract-v2-examples

## Research Question
Where do code examples and documentation live across the llama-cloud Python and TypeScript SDK repos? What needs updating when adding features like Extract V2?

## Summary
Both SDKs have 5 distinct documentation surfaces: README.md (manually maintained), api.md (auto-generated by Stainless), source code docstrings/JSDoc (auto-generated for base methods, manual for custom methods), examples/ directory (manually maintained, protected from generator), and CHANGELOG.md (auto-generated by release-please). The TypeScript SDK currently has better Extract V2 documentation coverage than Python, particularly in source code JSDoc examples. Both READMEs reference V1 extraction patterns in pagination examples.

## How It Works Today

### Documentation Flow
1. **OpenAPI spec changes** in Stainless Studio trigger codegen
2. **Stainless generator** produces: source code with docstrings, `api.md`, updates to types
3. **Generator protects**: `examples/` and `src/lib/` directories (CONTRIBUTING.md:37-39 in both repos)
4. **Custom methods** (run, waitForCompletion) are manually added and persist through regeneration via semantic merge
5. **Release-please** auto-generates CHANGELOG.md entries from conventional commits
6. **README.md** is manually maintained

### Documentation Surfaces Detail

**1. README.md** — Both repos, manually maintained
- Python: `llama-cloud-py/README.md` — Quick start, file uploads, async usage, error handling, retries, pagination
- TypeScript: `llama-cloud-ts/README.md` — Same structure, also includes MCP Server section
- Both list "Extract" in feature bullets (line 10)
- Pagination example in Python uses V1: `client.extraction.runs.list()` (README.md:140-150)
- Pagination example in TypeScript uses V2: `client.extract.list()` (README.md:114-135)

**2. api.md** — Both repos, auto-generated by Stainless
- Full API reference with type definitions and method signatures
- Python: 626 lines, has "# Extract" section (~line 86-100)
- TypeScript: 546 lines, has "# Extract" section (~line 75-100)
- No code examples — reference only

**3. Source Code Docstrings/JSDoc**
- Auto-generated methods (create, list, get, delete): Basic parameter descriptions, no examples
- Custom methods on TypeScript V2 extract.ts:
  - `waitForCompletion()` (lines 95-101): Full @example block
  - `run()` (lines 150+): Full @example block
- Custom methods on Python V2 extract.py:
  - `wait_for_completion()`: Detailed docstring with Args, Returns, Raises — but no Example block
  - `run()`: Detailed docstring with Args, Returns, Raises — but no Example block
- V1 extraction methods: Have examples in both SDKs (deprecation decorators present)
- Parsing methods: Have examples in both SDKs

**4. examples/ Directory** — Both repos, manually maintained (protected from generator)
- Organized by feature: classify/, extract/, index/, parse/, sheets/, split/
- Pattern: `<feature>_end_to_end.{py,ts}` + `<feature>_manual_polling.{py,ts}`
- V1 extract: 2 files per repo (extract_end_to_end, extract_manual_polling)
- V2 extract: 2 files per repo (extract_v2_end_to_end, extract_v2_manual_polling) — just added
- Python has 18 example files total, TypeScript has 17

**5. CHANGELOG.md** — Both repos, auto-generated by release-please
- Conventional Changelog format with version headers
- Entries derived from conventional commit messages
- Current Extract V2 entries: "api: api update" and "api: manual updates" (generic)

### Python Code Example Formatting
- `scripts/utils/ruffen-docs.py`: Validates/formats Python code blocks in markdown using ruff
- Run via `rye run format:docs` (pyproject.toml:66-73)

## External Best Practices

### Stainless Official Guidance
- Documentation should be built into the SDK via docstrings/JSDoc for IDE discovery
- README should include copy-pasteable code samples for common use cases
- OpenAPI spec descriptions flow into generated docstrings automatically
- Custom code in `examples/` and `src/lib/` persists through regeneration
- Bump.sh integration available for auto-injecting SDK code examples into API docs

### Industry Patterns (OpenAI, Anthropic SDKs — also Stainless-generated)
- Comprehensive README with installation, quickstart, streaming, error handling
- Inline docstrings with @example blocks in TypeScript
- examples/ directory with runnable scripts
- Helpers/convenience methods documented alongside raw API methods

### Where Our Repos Align
- Protected examples/ directory with feature-organized scripts
- Both end-to-end and manual polling example patterns
- README structure with quick start, error handling, retries, pagination
- TypeScript V2 extract has proper @example JSDoc blocks

### Where Our Repos Diverge
- Python V2 extract docstrings lack Example blocks (TypeScript has them)
- Python README pagination example still uses V1 API
- No Bump.sh integration for auto-injecting code samples into hosted docs
- No CI validation that example scripts are syntactically correct
- CHANGELOG entries use generic "api: manual updates" instead of descriptive messages
- No migration guide from V1 to V2 extraction

## Open Questions

1. **Should V1 extraction examples be removed or kept?** — V1 is deprecated but still functional. Keeping both may confuse users.
2. **Who owns the hosted docs at developers.llamaindex.ai?** — README links to external docs; unclear if those auto-update from the SDK repos.
3. **Should README be updated to show V2 extract examples?** — The pagination example still references V1 `client.extraction.runs.list()`.
4. **Is Bump.sh integration planned?** — Stainless supports it for auto-injecting code samples into API docs.
5. **Should we add CI validation for example scripts?** — Python has `ruffen-docs.py` for README but nothing validates example files compile/parse correctly.
6. **How to fix the "api: manual updates" changelog entry?** — Can we amend the commit on `next` before the release PR merges, or edit the release PR body directly?
